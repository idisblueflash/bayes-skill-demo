# Vibe Coding Hackathon 上，演示"贝叶斯选题 skill"的最佳 Demo 形式是什么？

## 结论

**采用 Live Demo + 实时贝叶斯可视化 + 观众参与**，而非 Web App 互动 PPT。具体操作建议：在台上现场运行 Claude Code skill，让观众喊出一个实际选题，把贝叶斯概率条的实时变化投影到屏幕上，全程控制在 6-8 分钟内。为防止 Live Demo 失败，提前备好一段已录制的回退视频（30 秒）并在本地缓存 API 响应，但绝不在开场就放视频。Hackathon 评委和观众看的是"AI 在工作"，而不是"PPT 在展示 AI 工作"——这个区别决定了哪种形式能制造真正的记忆点。

---

## 后验汇整

| 假说 | 初始先验 | 最终后验 | 变化方向 |
|------|---------|---------|---------|
| H₁ Live Demo + 实时可视化 + 观众参与 | 35% | 62% | ↑↑ |
| H₂ Web App 互动 PPT（预制叙事） | 35% | 19% | ↓↓ |
| H₃ 混合方案（PPT 打底 + 短片段 Live） | 20% | 14% | ↓ |
| H₄ 预录视频演示（完全规避 Live 风险） | 10% | 5% | ↓ |

## 论证过程

### 竞争假说设计

**主体**：Vibe Coding Hackathon demo 的呈现形式
**判断维度**：哪种形式最能让不一定关心写作的 Vibe Coding 观众，产生"原来还能这样用 AI"的感受

| 假说 | 内容描述 |
|------|---------|
| H₁ | Live Demo + 实时贝叶斯可视化 + 观众参与：现场运行 skill，观众提供选题，屏幕实时显示概率更新 |
| H₂ | Web App 互动 PPT（预制叙事）：预先设计好的 Web App，像 Prezi 一样控制节奏，视觉精美 |
| H₃ | 混合方案：PPT 或 Web App 打主结构，最后 2 分钟放一段真实 Live Demo 片段 |
| H₄ | 完全预录视频演示：录好整个流程视频，现场播放，规避所有 Live 风险 |

**初始先验设定逻辑**：演讲者最初倾向 H₂，主 Claude 建议 H₁，双方都有理由，故 H₁、H₂ 各取 35%，H₃ 作为折中方案取 20%，H₄ 因业界明显偏好 Live 而取 10%。

## 最终结论

### 核心结论（H₁，后验 62.2%）

**Live Demo + 实时贝叶斯可视化 + 观众参与**是最优 Demo 形式。最强支撑来自三条证据：
1. **Vibe Code Jam 真实案例**：从 100 人到 1400 人，现场 AI 表演就是 Vibe Coding 人群的原生格式
2. **观众参与制造记忆点**：让观众的选题进入演算，把旁观者变成共同创作者
3. **评委偏好真实运行**：台上几分钟的实时展示是"make or break moment"

### 残余不确定性（H₃，后验 5.7%）

**混合方案（PPT 打底 + 短片段 Live）**仍有 5.7% 支持度，原因是：
- Live 风险（网络/API）是真实存在的，混合方案提供了保底结构
- 叙事控制权论点有合理性，完全放弃控制有一定概率导致节奏失控
- **如果演讲者是第一次做 Hackathon 演讲、或对临场发挥缺乏信心**，H₃ 的实际优先级应提高

### 已排除假说

- **H₂（Web App 互动 PPT）**：降至 0.8%，被三条高区分度证据（观众参与 × Vibe Coding 原生格式 × 评委偏好 Live）联合证伪
- **H₄（完全预录视频）**：降至 0.0%，与 Vibe Coding 精神完全对立，无支撑

### 行动建议

1. **主方案选 H₁**：5-8 分钟内完成"问题设定（30秒）→ 观众提选题（20秒）→ 实时运行 skill（3-4分钟，屏幕显示概率条移动）→ 结论宣读（30秒）→ 一句话总结 skill 的迁移价值（30秒）"
2. **风险对冲三步**：本地缓存备份 API 响应 / 准备 30 秒回退视频（如 API 挂了当场切）/ 演讲前 30 分钟用同一环境跑一次完整流程
3. **跨界解释**：开场不要讲"写作选题"，改为"任何需要在多个方向中做选择的决策"，直接用 Vibe Coding 人群关心的案例（比如"要不要继续做这个 Hackathon 项目"）
4. **回退预案**：即使 Live 出错，演讲者的实时反应本身也是 Vibe Coding 精神的体现——"mistakes embraced as part of the art"

### 偏误声明

本分析的 7 条内部证据均来自演讲者/主 Claude 的框架设定，存在一定的**确认偏误风险**：支持 Live Demo 的证据比支持 PPT 的证据数量更多（5 vs 2）。外部证据 A（Vibe Code Jam）对 H₁ 的支持极强，但该案例更接近"表演型活动"，与"Hackathon 演示"场景有差异，可能轻微高估 H₁。将 H₁ 实际信心度理解为 55-65% 的区间更为稳健。
